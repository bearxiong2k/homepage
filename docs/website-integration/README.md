# Website Integration Dev Kit

## Purpose

This kit gives the repository a stable integration surface so the CIM paper atlas can become one project inside your personal website, and so future research atlases can follow the same pattern.

The current repo should remain a static Astro site. The atlas content source stays in Markdown frontmatter under `src/content/papers/`; taxonomy vocabulary stays in `src/data/taxonomy.json`; the main atlas route remains `/library/`; individual papers remain under `/papers/[slug]/`.

## Recommended architecture

Use a **static project-module architecture**:

```text
personal website shell
├── /                         personal homepage
├── /projects/                index of research/dev projects
├── /projects/cim-library/    narrative landing page for this atlas
├── /library/                 interactive CIM atlas
├── /papers/[slug]/           paper notes
└── /<future-project>/        future project modules
```

The atlas remains a first-class route instead of being squeezed into a tiny card. The homepage and project index should summarize it and link into `/library/` and representative papers.

## Integration contract

Every substantial project should expose three things:

1. **A project registry entry** in `src/data/project-registry.json`.
2. **A static export manifest** in `public/<project-id>.manifest.json`, generated by a script.
3. **A route contract** that says where the project landing page, interactive app, and detail pages live.

For CIM Library, the first export is generated by:

```bash
npm run export:atlas
```

The generated file is intended for your personal homepage, project cards, sitemap-like pages, or future cross-project dashboards.

## Development loop

Use this loop for every Codex session:

```bash
npm run qa
npm run validate
npm run export:atlas
npm run contract:website
npm run check
npm run build
```

For a quick baseline before dependencies are installed:

```bash
npm run smoke:website -- --fast
```

`--fast` skips Astro `check` and `build`, but still validates content, exports the manifest, and checks the website contract.

## What to build first

1. Add or polish `/` as your personal homepage.
2. Add `/projects/` and `/projects/cim-library/` using the project registry.
3. Keep `/library/` as the full atlas app.
4. Improve paper detail pages and provenance affordances.
5. Add the same registry/manifest pattern for future projects.

## Non-goals

Do not add a backend, database, PDF hosting, ranking score, coverage score, or hidden generated fields just for website integration. The public atlas should stay descriptive and static.
